# 企业级呼叫系统 - 第一阶段目标定义（Goals.md）

## 文档定位

本文件只定义**第一阶段做成什么样才算成功**，不重复技术栈、分层、目录结构等实现约束。

- 技术栈、架构、AI 执行铁律：以 `docs/SYSTEM_SPEC.md` 为准
- 任务拆解、执行顺序、每轮交付要求：以 `docs/PHASE1_TASKS.md` 为准
- 本文件聚焦第一阶段目标、非目标、验收口径

## 第一阶段目标

第一阶段的目标不是做完整 AICC，而是交付一个可启动、可验证、可继续迭代的 MVP，满足以下最小闭环：

1. 项目可以启动，核心模块结构完整，基础配置可用。
2. 可以通过 API 触发 FreeSWITCH 发起外呼。
3. 可以接收并处理 FreeSWITCH 呼入/呼出相关事件。
4. 通话结束后可以把核心通话日志写入 MySQL。
5. 可以通过查询 API 查看通话日志。
6. 提供最小化 Web 页面用于发起外呼和查看日志。

## 第一阶段非目标

第一阶段明确不做以下内容：

1. 坐席分配、技能组、排队、ACD。
2. IVR 导航、按键菜单、语音菜单。
3. 通话录音、录音下载、录音质检。
4. 报表大屏、复杂运营分析、实时监控中心。
5. 认证授权、多租户、RBAC/JWT。
6. 高可用、集群部署、深度性能优化。
7. 复杂前端交互，仅保留最小化可用页面。

## 阶段验收项

### G1 项目骨架可启动

**目标**：后端基础工程和公共模块可运行，后续功能开发有稳定落点。

**通过标准**：
1. `backend/call-center` 多模块工程结构完整。
2. `common` 与 `call-core` 模块职责清晰。
3. 项目在本地可启动，不因 FreeSWITCH 未连接而直接崩溃。
4. MySQL 配置存在且连接失败时有明确错误信息。
5. 统一响应格式和全局异常处理可用。

### G2 外呼链路可验证

**目标**：从接口到 FreeSWITCH 的 outbound 链路可走通。

**通过标准**：
1. `POST /api/call/outbound` 可接收 `caller`、`callee` 参数。
2. 参数非法时返回统一错误响应。
3. FreeSWITCH 可用时，接口能触发真实外呼。
4. 外呼发起后能产生可追踪的 `callId` 或等价标识。
5. 至少能在日志中确认外呼发起结果与失败原因。

### G3 呼入/事件接入可验证

**目标**：系统可以接收并处理 FreeSWITCH 事件，而不是只有“发起呼叫”能力。

**通过标准**：
1. FreeSWITCH 呼入或通话生命周期事件可进入 `call-core`。
2. 关键事件至少覆盖呼叫创建和呼叫结束。
3. 事件解析失败时记录日志，但不阻断主流程。
4. 事件发布遵守 `EventPublisher` 抽象，不绕过既定约束。

### G4 通话日志可落库可查询

**目标**：通话数据不是只在日志里出现，而是能沉淀到数据库并被查询。

**通过标准**：
1. 提供幂等建表脚本，如 `schema_v1.sql`。
2. 通话结束后可写入一条通话记录。
3. 日志至少包含 caller、callee、callId、开始时间、结束时间、状态、hangup cause 等关键字段。
4. 时长字段有明确计算规则，并能在代码或文档中追溯。
5. `GET /api/call/log` 可按手机号和时间范围查询。

### G5 最小化 Web 可用

**目标**：提供最基础的学习和演示界面，验证前后端联动。

**通过标准**：
1. 页面可发起外呼请求。
2. 页面可查看通话日志列表。
3. 接口成功和失败状态有基本反馈。
4. 不要求复杂样式和高级交互，但必须能完成基本操作。

## 总体验收标准

第一阶段完成时，应至少满足以下条件：

1. 项目可以在本地启动并完成基础联调。
2. 外呼 API、日志查询 API 可访问且返回统一格式。
3. 至少一条真实或可明确说明的降级通话链路被验证。
4. MySQL 中能看到通话日志数据，且数据字段基本完整。
5. 文档能说明哪些能力已经真实打通，哪些仍是 mock / 降级。
6. 若实现偏离原计划，偏差已回写到相关文档。
