# 第二阶段目标定义（PHASE2_GOALS.md）

## 文档定位

本文档定义第二阶段“做成什么样才算成功”，不重复技术栈、实现边界和阶段一已完成内容。

- 技术栈、架构边界、阶段规划：以 `docs/SYSTEM_SPEC.md` 为准
- 第二阶段任务拆解与执行顺序：以 `docs/PHASE2_TASKS.md` 为准
- 第一阶段 MVP 目标与验收：仍以 `docs/GOALS.md` 为准

## 第二阶段目标

第二阶段的目标不是一次做完整 AICC，而是在第一阶段“可呼叫、可记日志、可查询”的基础上，补齐“受管呼叫、坐席归属、AI 通话骨架”三类能力，形成可继续扩展的人机协同版本。

第二阶段至少应满足以下闭环：

1. 系统只处理“属于本系统管理范围”的呼叫，而不是 FreeSWITCH 全量事件直接入库。
2. 支持坐席账号、分机号、坐席状态的基础管理与绑定。
3. 支持对受管分机的呼入/呼出做明确归属，减少 `voicemail`、扫描流量等脏数据干扰。
4. 建立 AI 通话引擎抽象，打通 ASR/TTS/LLM 的最小骨架，但允许先用 mock/provider stub 验证链路。
5. 支持一次最小化的人机协同流程，例如 AI 接待、转人工、结束通话。
6. Web 页面可查看坐席、分机绑定关系和受管通话的基础状态。

## 第二阶段非目标

第二阶段不做以下内容：

1. 完整 IVR 编排平台。
2. 录音存储、录音下载、录音质检。
3. 复杂 ACD 排队策略、多技能组智能分配。
4. 运营大屏、复杂 BI 报表、实时监控中心。
5. 多租户、RBAC/JWT、复杂权限体系。
6. 生产级高可用、集群容灾、深度性能压测。

## 阶段验收项

### G1 受管呼叫边界清晰

**目标**：系统只落库和展示属于本系统管理范围的呼叫。

**通过标准**：
1. 存在“坐席/分机绑定”或等价白名单机制。
2. 默认不再将 FreeSWITCH 所有事件直接落库。
3. 可过滤明显非业务数据，如 `voicemail` 内部腿、异常号码、扫描流量。
4. 查询接口和前端列表展示的都是受管通话数据。

### G2 坐席基础能力可用

**目标**：系统具备第二阶段最小坐席模型。

**通过标准**：
1. 可维护坐席基本信息。
2. 可维护坐席与分机号绑定关系。
3. 可表示坐席状态，如空闲、忙碌、离线。
4. 页面可查看或操作这些基础数据。

### G3 AI 通话骨架可跑通

**目标**：建立 AI 语音能力接入点，而不是把通话能力继续写死在 FreeSWITCH 适配层。

**通过标准**：
1. 存在 ASR、TTS、LLM 的抽象接口。
2. 可通过 mock/provider stub 验证调用链路。
3. 至少存在一条最小 AI 会话流程定义。
4. AI 相关代码不直接污染 Controller/Mapper 层。

### G4 人机协同链路成立

**目标**：系统从“单纯会打电话”升级为“能进入 AI 或人工处理流程”。

**通过标准**：
1. 至少支持一种 AI 接待入口。
2. 至少支持一种转人工或人工接管动作。
3. 通话状态和处理结果可追踪。
4. 关键过程有日志或事件可验证。

### G5 第二阶段交付可验证

**目标**：第二阶段不是概念设计，而是可启动、可操作、可验证。

**通过标准**：
1. 后端可启动。
2. 第二阶段新增 API 可访问且返回统一格式。
3. 前端可完成至少一条新增业务流程。
4. 文档说明哪些能力是真实打通，哪些仍是 mock/stub。
5. 第二阶段任务文档包含执行记录与偏差记录。

## 总体验收标准

第二阶段完成时，应至少满足以下条件：

1. 受管呼叫模型落地，系统不再无差别消费全部 FS 事件作为业务数据。
2. 坐席/分机基础管理闭环可用。
3. AI 能力接入层抽象完成，存在最小对话流程闭环。
4. 人机协同链路至少一条可验证。
5. 页面和接口都能展示第二阶段新增能力。
6. 文档、代码、验证结果保持同步。
