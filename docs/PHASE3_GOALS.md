# 第三阶段目标定义（PHASE3_GOALS.md）

## 文档定位

本文档定义第三阶段“做成什么样才算成功”，不重复技术栈、总体架构和前两阶段已完成内容。

- 技术栈、架构边界、分期路线图：以 `docs/SYSTEM_SPEC.md` 为准
- 第三阶段任务拆解与执行顺序：以 `docs/PHASE3_TASKS.md` 为准
- 第二阶段完成基线与当前能力现状：以 `docs/PHASE2_TASKS.md` 为准

## 第三阶段目标

第三阶段的目标不是直接做成“生产级完整呼叫中心”，而是在第二阶段“受管呼叫 + 坐席归属 + 最小 AI 流程”基础上，补齐真正可运营的人机协同链路，重点覆盖以下四类能力：

1. 把第二阶段的 mock/stub 闭环推进到至少一条真实 FreeSWITCH 联调链路。
2. 把“转人工”从最小状态流转升级为“可接、可追踪、可回退”的受管接管流程。
3. 建立录音与通话摘要的最小闭环，让一次通话结束后有可查询、可回放、可总结的结果沉淀。
4. 引入最小 IVR/场景路由入口，让系统能在 AI、人工、按键导航之间做明确分流。

第三阶段至少应满足以下闭环：

1. 受管来电或 AI 外呼进入系统后，能够基于场景或 IVR 进入 AI 或人工处理链路。
2. AI 无法完成处理时，可无缝转人工，且人工接管结果、失败原因、降级动作可追踪。
3. 通话结束后，系统能落录音元数据，并产出可查询的摘要或结果记录。
4. 前端页面可查看第三阶段新增的关键状态，包括转人工过程、录音信息、通话摘要。
5. 至少一条真实 provider/真实链路联调结果被明确记录，和 mock 验证结果显式区分。

## 第三阶段非目标

第三阶段不做以下内容：

1. 完整技能组路由、复杂 ACD 排队和预测式外呼。
2. 全量实时坐席辅助能力，如逐句提示词推荐、知识库自动检索编排。
3. 完整 IVR 平台设计器、可视化流程编排和多分支复杂流程编辑。
4. 录音质检、质检评分、运营 BI 大屏、复杂统计报表。
5. 多租户、完整 RBAC、生产级审计合规和高可用集群改造。
6. 所有 AI provider 的一次性全面接入，只要求至少具备可切换抽象和一条真实验证链路。

## 阶段验收项

### G1 真实链路联调成立

**目标**：第三阶段不再只停留在 mock 骨架，而是至少验证一条真实语音或呼叫链路。

**通过标准**：
1. 至少存在一条真实 FreeSWITCH 相关联调记录，且能够说明入口、事件、结果。
2. 至少存在一类真实 AI provider 或真实语音能力切换验证结果。
3. 文档中明确区分“真实打通”“本地 mock”“接口占位”三类状态。
4. 联调失败场景有明确降级方案，而不是静默忽略。

### G2 人机协同链路可运营

**目标**：转人工不只是改状态，而是形成“可接入、可失败、可追踪”的运营闭环。

**通过标准**：
1. AI 转人工时可明确定位目标坐席或目标分机。
2. 人工接管结果可区分成功、失败、超时、取消等状态。
3. 转人工关键过程有日志、事件或会话状态可查询。
4. 无可用坐席时有明确兜底动作，例如回退话术、结束通话或重新排队占位。

### G3 录音与摘要闭环可用

**目标**：系统具备第三阶段最小通话沉淀能力。

**通过标准**：
1. 通话录音至少落元数据，包含 `callId`、录音路径或资源标识、时长、状态。
2. 前端或接口可查询录音信息。
3. 通话结束后可生成最小摘要结果，至少包含摘要正文、处理结果或关键标签之一。
4. 录音和摘要能力明确说明哪些是真实链路，哪些仍为 stub。

### G4 IVR/场景分流入口成立

**目标**：系统可以在 AI、人工、按键导航之间做最小化分流，而不是所有通话都走单一路径。

**通过标准**：
1. 至少支持一个最小 IVR 或场景路由入口。
2. 至少支持按键或规则分流到 AI、人工或结束通话中的两类以上目标。
3. 分流结果与 `CallSession` 或等价模型可关联。
4. 路由过程失败时有默认去向或降级结果。

### G5 第三阶段交付可验证

**目标**：第三阶段交付物可启动、可演示、可验证，而不是停留在方案层。

**通过标准**：
1. 后端可启动，第三阶段新增 API 可访问且返回统一 `Result` 结构。
2. 前端至少可完成一条第三阶段新增业务流程。
3. 至少存在一组覆盖第三阶段核心服务的自动化测试。
4. 第三阶段任务文档包含执行记录、偏差记录和联调状态说明。
5. 明确记录当前仍未进入第三阶段范围的能力，避免验收时边界漂移。

## 总体验收标准

第三阶段完成时，应至少满足以下条件：

1. 第二阶段的人机协同链路被升级为至少一条真实可联调的运营链路。
2. 转人工流程具备可追踪的成功/失败/降级结果。
3. 录音与摘要能力形成最小数据闭环，并能被查询或展示。
4. 系统具备最小 IVR 或场景分流入口。
5. 页面和接口都能展示第三阶段新增能力。
6. 文档、代码、联调结果、测试结果保持同步。

## AI 使用说明

当 AI 读取本文档时，应将其理解为“第三阶段是否完成”的判断依据，而不是重新定义第二阶段范围。

- 第二阶段已完成能力优先参考 `docs/PHASE2_TASKS.md`
- 第三阶段目标判断优先参考本文档
- 若与 `docs/SYSTEM_SPEC.md` 或 `docs/PHASE3_TASKS.md` 冲突，以后两者为准
